0.1

LLM의 본질 — 입력과 출력, 그게 전부다

claude-opus-4-6, gpt-4o, gemini — 이름이 뭐든 간에, 이들의 본질은 단 하나입니다.

시작은 이랬습니다
텍스트 입력
→
LLM
→
텍스트 출력

처음엔 정말 이게 전부였습니다. 텍스트를 받아서, 텍스트를 뱉는 것.

지금은 여기서 더 발전했습니다. 이미지를 보고 이해하고, 음성을 듣고, 이미지를 직접 생성하기도 합니다. 이른바 멀티모달(Multimodal) — 텍스트뿐 아니라 다양한 형태의 데이터를 다루게 된 것이죠.

하지만 본질은 달라지지 않았습니다. 입력의 종류가 다양해지고, 출력의 종류가 다양해졌을 뿐 — 여전히 "받아서 뱉는" 구조입니다.

죽었다 깨어나도 웹 검색을 못 합니다. 코드를 실행할 수 없습니다. 파일 하나 읽을 수 없습니다. 이메일도 못 보내고, API도 못 호출합니다. 아무리 멀티모달이 되어도, LLM은 입력을 받아 출력을 생성하는 변환기일 뿐입니다.

"그런데 ChatGPT는 웹검색도 하고 코드도 실행하잖아요?" — 맞습니다. 하지만 그건 LLM이 한 게 아닙니다. LLM 주변에 만들어둔 시스템이 한 겁니다. 이 차이가 이 글의 핵심입니다.

LLM은 학습 데이터에서 패턴을 익힌 뒤, 주어진 입력의 다음 토큰(단어 조각)을 예측하는 방식으로 작동합니다. 원리만 놓고 보면 "그냥 다음 단어 맞추기"입니다. (물론 그 위에 인간 피드백 학습(RLHF) 등 다양한 추가 훈련이 더해져서 "다음 단어 예측기"가 "지시를 따르는 어시스턴트"로 다듬어집니다.)

다만, 그게 너무나도 — 정말 경이로울 정도로 — 잘 됩니다. 수학 문제를 풀고, 코드의 논리적 오류를 찾아내고, 외국어를 번역하고, 사진 한 장을 보고 상황을 파악하고, 복잡한 요구사항을 이해해서 구조화된 결과물을 만들어냅니다. "다음 토큰 예측"이라는 단순한 원리에서 이런 능력이 나온다는 것 자체가 현대 AI의 가장 놀라운 발견 중 하나입니다.

"그런데 대화하다 보면 이전 내용을 기억하는 것 같은데요?" — 그것도 사실 같은 원리입니다. LLM은 대화를 "기억"하는 게 아닙니다. 매번 응답할 때마다, 지금까지의 대화 내용 전체를 입력으로 다시 넣어주는 겁니다.

→
민수님요!
↑ 입력이 매 턴마다 점점 길어지는 게 보이시나요? 기억하는 게 아니라, 매번 전부 다시 넣는 겁니다.

LLM에게 "기억"이란 없습니다. 그냥 매번 전체 대화를 처음부터 다시 읽는 것입니다. 이래서 대화가 길어지면 느려지거나, 아주 오래 전에 했던 말을 "잊어버리는" 것처럼 보이기도 합니다. 입력에 넣을 수 있는 양에 한계가 있으니까요. 이 한계를 컨텍스트 윈도우라고 부릅니다.

"그러면 메모리 기능은 뭐예요? 대화 끝나도 기억하던데요?"

ChatGPT나 Claude에 있는 메모리도 마법이 아닙니다. 과정을 뜯어보면 이렇습니다:

"메모리"의 실체
STEP 1 — 대화 중
사용자: "나는 백엔드 개발자고 Python을 주로 써"
LLM이 판단: "이 정보는 나중에도 유용하겠다"
→ LLM이 "이건 저장할 만하다"는 신호를 출력
→ 외부 시스템이 이 신호를 받아서 어딘가(파일, DB)에 저장
STEP 2 — 며칠 뒤, 새 대화 시작
외부 시스템이 저장된 메모리를 읽어옴
→ 사용자 메시지와 함께 LLM 입력에 슬쩍 끼워넣음:
📋 [저장된 메모리] 사용자는 백엔드 개발자, Python 선호
📄 [시스템 프롬프트] 당신은 도움이 되는 어시스턴트입니다...
💬 [사용자] API 하나 만들어줘
↑ LLM은 이 전체를 하나의 입력으로 받음

핵심을 짚어보면 — LLM은 "이건 저장할 만하다"고 판단했을 뿐입니다. 실제로 저장한 건 외부 시스템이고, 다음에 다시 꺼내서 입력에 넣어준 것도 외부 시스템입니다. LLM 자체는 여전히 아무것도 기억하지 못합니다. 매번 누군가가 커닝페이퍼를 같이 넣어주는 것일 뿐이죠.

결국 본질은 변하지 않습니다. 아무리 뛰어나도, LLM 혼자서는 외부 세계와 상호작용할 수 없습니다. 입력을 받아 출력을 만들 뿐. 이 한계를 어떻게 뛰어넘었는지가 다음 이야기입니다.

0.2

구조화된 출력 — Tool의 탄생

전환점은 LLM이 구조화된 형태의 텍스트를 안정적으로 출력할 수 있게 되면서 찾아왔습니다. 자연어 대신 JSON 같은 정해진 포맷으로 출력하게 된 거죠.

// LLM이 이런 걸 출력할 수 있게 되었다 { "tool": "web_search", "arguments": { "query": "오늘 서울 날씨" } }

이 순간 게임이 바뀌었습니다. LLM이 직접 웹검색을 하는 건 여전히 불가능하지만, "이 함수를 이 파라미터로 호출해줘"라고 요청할 수 있게 된 겁니다. 이것이 Tool Use, 또는 Function Calling의 시작입니다.

Tool Use의 작동 방식
텍스트
→
LLM
→
{ tool_call }
→
외부 시스템 실행
실행 결과
→
LLM
→
최종 응답

핵심을 짚어보면:

① LLM은 여전히 텍스트만 출력합니다. 다만 그 텍스트가 "이 도구를 실행해달라"는 구조화된 요청인 거죠.

② 실제 실행은 외부 시스템이 합니다. 웹검색을 하는 건 검색 API고, 코드를 돌리는 건 샌드박스 런타임입니다.

③ 결과가 다시 LLM에게 돌아옵니다. LLM은 이 결과를 바탕으로 사용자에게 자연어 응답을 만듭니다.

앞에서 메모리를 설명할 때, LLM이 "이건 저장할 만하다"는 신호를 출력한다고 했죠? 그 신호의 정체가 바로 이겁니다:

// 메모리 저장도 결국 Tool 호출이었던 것 { "tool": "save_memory", "arguments": { "content": "사용자는 백엔드 개발자, Python 선호" } } // → 외부 시스템이 이걸 받아서 실제로 파일이나 DB에 저장한다 // → LLM은 JSON을 출력했을 뿐, 저장 자체는 한 적이 없다

비유하자면 — LLM은 팔다리가 없는 두뇌입니다. Tool은 그 두뇌에 연결된 팔과 다리죠. 두뇌가 "오른손으로 컵을 집어"라고 신호를 보내면, 팔이 실행합니다. 두뇌 자체가 컵을 집는 건 아닙니다.

0.3

MCP의 등장 — 도구의 USB 포트

Tool이 유용하다 보니 모두가 만들기 시작했습니다. 문제는 표준이 없었다는 겁니다.

누군가 Slack 연동 Tool을 만들면, 그건 특정 플랫폼에서만 작동했습니다. 다른 플랫폼에서 쓰려면 매번 새로 만들어야 했죠. 마치 USB가 발명되기 전, 프린터·키보드·마우스가 각각 다른 포트를 쓰던 시절과 같았습니다.

그래서 Anthropic이 MCP (Model Context Protocol)를 만들었습니다. Tool을 표준화된 방식으로 만들고, 공유하고, 재사용할 수 있게 하는 프로토콜입니다.

무엇이든
↕

↕

위쪽이 MCP 서버(Tool을 제공하는 서비스들), 아래쪽이 MCP 클라이언트(LLM을 사용하는 앱들)입니다.

MCP가 가운데서 이 둘을 이어줍니다. USB 포트처럼요. 누군가 Slack MCP 서버를 하나 만들어두면, Claude Code에서도, Cursor에서도, 내가 만든 앱에서도 그냥 꽂아서 쓸 수 있습니다.

MCP 이전: "이 Tool을 내 앱에서 쓰려면 어떻게 연결하지…?"
MCP 이후: "MCP 서버 주소 넣으면 끝."

0.4

파일시스템 + 명령어 실행 — 에이전트의 탄생

날씨 검색 Tool, 이메일 전송 Tool, 번역 Tool — 이렇게 하나하나 만들어서 쥐어주는 방식도 유용했지만, 한계가 있었습니다. 새로운 기능이 필요할 때마다 누군가 Tool을 만들어야 했으니까요.

그러다 결정적인 조합이 등장합니다. LLM에게 파일 읽기/쓰기 Tool과 터미널 명령어 실행 Tool을 쥐어준 것입니다.

이 두 개면 끝입니다. 터미널 명령어를 실행할 수 있다는 건, curl로 API를 호출하고, npm install로 패키지를 설치하고, git commit으로 코드를 저장하고, python으로 스크립트를 돌릴 수 있다는 뜻입니다. 일일이 Tool을 만들어줄 필요 없이, LLM이 알아서 할 수 있게 된 것입니다. 이 순간, LLM은 그야말로 날아다니기 시작했습니다.

코딩 에이전트의 작동 루프
사용자: "이 버그 좀 고쳐줘"
↓
LLM이 상황 판단
↓
Tool: 파일 읽기 📄
↓
LLM이 코드 분석
↓
Tool: 파일 수정 ✏️
↓
Tool: 테스트 실행 ▶
↓
LLM: 통과했나? → 반복 or 완료
반복 (agentic loop)
↓
"버그 수정 완료, 테스트 통과 ✓"

이게 바로 Claude Code 같은 코딩 에이전트의 작동 방식입니다.

코드를 읽고 → 이해하고 → 수정하고 → 테스트를 돌리고 → 실패하면 다시 고치고 → 성공하면 커밋까지. 이 모든 과정을 LLM이 스스로 판단하며 반복합니다. 사람이 하던 것과 동일한 루프를 도는 거죠.

여기서 꼭 짚고 넘어가야 할 게 있습니다. LLM이 {"tool": "read_file", ...} 같은 구조화된 출력을 하면, 누군가는 그걸 받아서 진짜로 실행해줘야 합니다.

그 "누군가"가 바로 호스트 시스템입니다. Claude Code의 경우, 그게 여러분의 컴퓨터입니다. LLM이 {"tool": "read_file"}을 출력하면, 여러분의 컴퓨터가 실제로 파일을 열어서 내용을 LLM에게 돌려줍니다. {"tool": "bash", "command": "npm test"}를 출력하면, 여러분의 터미널에서 진짜로 실행됩니다.

ChatGPT라면? OpenAI의 서버가 대신 실행합니다. 웹의 Claude라면? Anthropic의 서버가 실행합니다. LLM은 항상 "이 Tool을 이 파라미터로 실행해달라"는 구조화된 출력을 할 뿐이고, 실행은 언제나 다른 누군가의 몫입니다.

그래서 에이전트의 공식은 이렇습니다: "도구를 쓸 줄 아는 LLM + 도구를 실행해주는 호스트 + 적절한 도구 세트 = 에이전트". LLM이 판단하고, 호스트가 실행하고, 결과가 다시 돌아오고, 이 루프가 자율적으로 돌아가는 것이 에이전트입니다.

0.5

Skills의 시대 — 에이전트의 두뇌를 확장하다

에이전트가 잘 작동하려면 Tool 외에도 많은 지침이 필요합니다. "커밋할 때는 이렇게 해라", "우리 팀 API 규칙은 이거다", "배포 절차는 이 순서로" 같은 것들. 처음엔 이걸 전부 하나의 거대한 프롬프트에 넣었습니다.

문제는 — 0.1에서 봤듯이 — LLM에게 넣을 수 있는 입력에는 한계가 있다는 겁니다. 컨텍스트 윈도우는 유한하고, 거기에 "혹시 몰라서" 모든 지침을 때려넣으면 정작 중요한 대화 내용이 들어갈 자리가 줄어듭니다. 배포 지침이 필요 없는 상황에서도 배포 매뉴얼이 컨텍스트를 차지하고 있는 거죠.

그래서 나온 게 Skill입니다. 에이전트는 평소에 각 Skill의 이름과 짧은 설명만 알고 있다가, 실제로 필요한 순간에만 해당 Skill의 내용을 로드합니다. 컨텍스트를 낭비하지 않으면서도, 필요할 때는 전문적인 지침을 바로 꺼내 쓸 수 있는 구조입니다.

Skill의 실체는 폴더입니다. 그 안에 지침 파일(SKILL.md), 템플릿, 스크립트, 코드 파일, 참고 문서 등이 들어있고, 에이전트가 필요할 때 필요한 파일을 쏙쏙 꺼내 씁니다.

# Skill 하나의 구조 — 결국 폴더입니다 my-skill/ ├── SKILL.md # 주요 지침 (이것만 필수) ├── template.md # 템플릿이 될 수도 있고 ├── examples/ │ └── sample.md # 예제 모음이 될 수도 있고 └── scripts/ └── validate.sh # 스크립트가 될 수도 있고... 뭐든 가능

실제 SKILL.md 파일은 이렇게 생겼습니다:

# 파일: .claude/skills/deploy/SKILL.md --- name: deploy description: 프로덕션에 애플리케이션 배포 disable-model-invocation: true ← Claude가 멋대로 배포 못하게 --- $ARGUMENTS를 프로덕션에 배포합니다: ← /deploy staging 하면 $ARGUMENTS = "staging" 1. 테스트 스위트 실행 2. 애플리케이션 빌드 3. 배포 타겟에 푸시 4. 배포 성공 여부 확인

이 파일 하나로 에이전트는 /deploy라는 새로운 능력을 얻습니다. Skill은 이렇게 누구나 만들 수 있고, 공유할 수 있고, 꽂으면 바로 작동합니다.

더 강력한 Skill은 스크립트를 번들하거나, 실행 전에 외부 데이터를 주입하거나, 별도 에이전트를 띄워서 격리된 환경에서 실행하기도 합니다. 단순한 텍스트 지침이 아니라, 에이전트의 능력을 근본적으로 확장하는 모듈인 거죠.

비유를 바꿔봅시다.

Tool이 손과 발이라면,
Skill은 기술과 경험입니다.

같은 손을 가져도 목수의 손과 프로그래머의 손은 다릅니다. 도구가 같아도 어떤 Skill이 장착되어 있느냐에 따라 에이전트의 능력은 완전히 달라집니다. 그리고 이 Skill은 마크다운 파일 하나로 만들고 공유할 수 있습니다.

0.6

하네스 — 부품이 아니라 완성품

지금까지 부품을 하나씩 살펴봤습니다. Tool, MCP, 에이전트 루프, Skill, 그리고 이 모든 걸 실행해주는 호스트 시스템까지.

그런데 실제로 쓸 수 있는 에이전트를 만들려면, 이 부품들을 하나로 조립해야 합니다. 어떤 Tool을 쓸 수 있게 할지, 어떤 Skill을 언제 로드할지, 컨텍스트 윈도우가 가득 차면 어떻게 할지, 에이전트가 위험한 행동을 하려 하면 어떻게 막을지, 세션이 끊기면 어떻게 이어갈지 — 이런 것들을 전부 설계하고 관리하는 시스템이 필요합니다.

이것을 하네스(Harness)라고 부릅니다.

Claude Code는 하나의 하네스입니다. Cursor도 하네스고, 여러분이 Agent SDK로 직접 만드는 것도 하네스입니다. 같은 LLM이라도 어떤 하네스에 담기느냐에 따라 전혀 다른 에이전트가 됩니다.

그리고 Claude Code는 이 하네스를 여러분이 직접 설정할 수 있게 열어두었습니다. CLAUDE.md로 프로젝트 규칙을 정하고, Skill을 만들어 워크플로우를 추가하고, MCP 서버를 연결해 외부 서비스를 붙이고, Hook으로 자동화를 걸고, 권한 설정으로 가드레일을 조절합니다. 즉 Claude Code는 완성된 하네스이면서 동시에, 여러분의 필요에 맞게 커스터마이징할 수 있는 하네스입니다.

이 용어는 2025년 11월 Anthropic의 Justin Young이 "Effective harnesses for long-running agents"에서 AI 에이전트 맥락에서 언급했고, 이후 OpenAI, LangChain, Salesforce 등도 같은 용어를 쓰면서 업계 전반에 퍼졌습니다. 아직 논쟁 중이긴 합니다 — "모델이 더 좋아지면 하네스는 필요 없어진다"는 반론도 있고, "그냥 기존 인프라 엔지니어링의 리브랜딩 아니냐"는 시각도 있습니다. 하지만 현재로서는 이 개념을 설명하는 가장 널리 쓰이는 용어입니다. (작성자 개인적으로는 개념은 좋지만 용어 자체가 너무 인간 중심적이라 AI 감수성이 부족하지 않나 싶...)

진화의 전체 그림:

입력 → 출력 (순수 LLM)
→ + 구조화된 출력 (Tool 호출 가능)
→ + MCP (표준화된 도구 생태계)
→ + 파일시스템 & 명령어 실행 (에이전트)
→ + Skill (에이전트에게 전문성을 부여하는 확장 모듈)
→ + 하네스 (이 모든 걸 조립한 완성품)

LLM 자체는 한 번도 변한 적 없습니다. 여전히 입력을 받아 출력을 만들 뿐입니다. 변한 건 LLM을 둘러싼 세계입니다.
